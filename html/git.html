Перед вами небольшая статья-шпаргалка, которая включает в себя удобные, но не всем знакомые команды Git, которые могут стать вашими постоянными спутниками в разработке.

1. git init

Эта команда создаёт новый репозиторий Git в пустой директории или превращает существующую директорию в новый репозиторий Git.


2. git clone

Клонирует репозиторий из удаленного хранилища на локальную машину

Если при клонировании задать --depth= "Число", то git log покажет "Число +1" ревизий: последняя ревизия клонируемой ветки (по умолчанию master) + "Число" история коммитов, усечённая до последних ревизий.

git clone --depth=1 git://someserver/somerepo dirformynewreporm -rf !$/.git


3. git remote add

Добавить удалённый репозиторий можно параметром remote add, указав shortname и url требуемого репозитория.

Пример: git remote add origin https://github.com/MrKrishnaAgarwal/Git-CheatSheet.git


4. git remote -v

Эта команда используется для просмотра подключенных удаленных репозиториев.


5. git status

Эта команда используется для просмотра статуса файлов в вашем локальном репозитории. Отслеживаются ли файлы? Не отслеживается? Изменены ли они?


6. git add

Добавить отдельный файл в область подготовленных файлов можно параметром add с указанием имени файла.

Пример: git add index.html

git add index.html style.css

Кроме того, можно добавить все файлы и папки в эту область, предоставив wildcard . вместо имени файла:

git add .


7. git reset

Эта команда используется для мягкой или жёсткой отмены изменений, позволяет сбросить состояние проекта до нужного коммита в локальной директории (отмена неотправленного коммита).

Например : git reset --soft HEAD^


8. git revert

Для отмены отправленного коммита используется reset ( возврат к состояние коммита 7880ae2 )

Например : git revert --soft 7880ae2


9. git push

Отправлять изменения в удалённый репозиторий можно параметром push с указанием имени репозитория и ветки.

Например: git push репозиторий ветка


10. git push -u ...

Ключ -u (полный вариант --set-upstream) создаёт в удалённом репозитории ветку, соответствующую локальной и связывает их.

Пример: git push -u origin master

Эта команда используется для отправки закоммиченных файлов в удаленный репозиторий (также известный как GitHub) в указанной ветке. Используйте эту команду, когда вы впервые отправляете файлы в удаленный репозиторий. Он зафиксирует место, куда вы отправляете эти файлы. И в следующий раз можно будет использовать команду git push.


11. git commit

Cохранение изменений в Git.

Например: git commit -m “Text message”

Команда -m позволяет указать commit message без обращения к редактору

git commit -m "added navigation bar"

Пример: git commit --amend

Команда --amend вносит в предыдущий коммит изменения, которые подготовлены к коммиту. Используется и для редактирования предыдущего commit message, если в нём допущены ошибки.


12. git fetch

Эта команда используется для получения самой последней версии удалённого репозитория. Загружает коммиты, файлы и ссылки из удаленного репозитория в ваш локальный репозиторий.


13. git pull

Эта команда используется для извлечения только что полученной информации и ее загрузки в локальный репозиторий. git pull запускает немедленное обновление локального репозитория.


14. git branch "имя ветки"

Эта команда используется для создания новой ветки.

git branch используется для просмотра всех веток в репозитории.

git branch -a используется для просмотра всех веток удаленного репозитория.


15. git checkout

Эта команда используется для переключения на ранее созданные ветки.

Пример: git checkout master или git checkout develop

git checkout -b "Имя новой ветки" — ещё один вариант использования, при котором происходит создание новой ветки и переход к ней.


16. git merge

Объединить две ветки можно параметром merge с указанием имени ветки. Команда объединит указанную ветку с текущей.

Пример: git merge develop

Здесь текущая ветка наследует код из ветки разработки.


17. git merge — abort

Эта команда используется для отмены слияния.

Если нет ошибок, слияние будет успешным. Следовательно, это прерывание можно использовать только в ситуациях, когда слияние не удалось. Как понять, что нужно использовать эту команду? Ваш терминал скажет, что слияние не удалось. Он также может предложить вам исправить конфликты слияния.

Вот ещё один признак: ~/NextCloud/Documents/Web Projects/Cloud4Y (master)

Посмотрите в самый конец строки. В скобках написано (master). Это потому, что мы находимся в основной ветке. Если вы находитесь в ветке разработки, значение будет (develop). Если смерджить не получилось, появится надпись (master|merging) или что-то в этом роде.


18. git merge -X theirs

Эта команда используется для объединения двух веток. И если есть конфликты слияния, эта команда просто предположит, что вы предпочитаете изменения, сделанные в указанной ветке (а не в текущей).

Пример: git merge -X theirs develop


19. git config — это удобная функция, которая используется для настройки значений конфигурации Git на глобальном и локальном уровнях проекта. Примеры команд:

git config --local user.name "Name"

git config --local user.email "your_email@example.com"

 git config --global user.name "Name"

 git config --global user.email "your_email@example.com"


20. git diff

Можно просматривать список изменений, внесённых в репозиторий, используя параметр diff. По умолчанию отображаются только изменения, не подготовленные для фиксации.


21. git rebase

Перебазирование — это процесс объединения или перемещения последовательности коммитов на новый родительский снимок.

git rebase -i HEAD~N

Склеить коммиты, переписав историю с момента HEAD~N, т.е. с того, что было N коммитов назад. -i — означает в интерактивном режиме.


23. git stash

Эта команда используется для сохранения неподтверждённых изменений в отдельном хранилище, чтобы можно было вернуться к ним позже. Сами файлы возвращаются к исходному состоянию. Команда полезна, когда вы работаете над одной веткой, хотите переключиться на другую, но вы ещё не готовы сделать коммит в текущей ветке. Таким образом, вы прячете изменения в коде, переключаетесь на другую ветку, возвращаетесь к исходной ветке, а затем разархивируете свои изменения.

git stash pop позволяет применить ранее отложенные изменения.
